MSU Curriculum Mapping & Dependency Visualization

Built a system to parse, clean, and visualize course prerequisite data at Michigan State University as a directed graph. The tool reveals curriculum structure, bottleneck courses, and student pathways, with a functional API and a web-based UI prototype.

Python pandas NetworkX Plotly matplotlib Jupyter Notebooks Regular Expressions (regex)
More details

Problem

MSU's raw course prerequisite data is messy and unstructured, making it difficult to understand how courses connect across a degree program. Advisors and researchers lacked a clear way to visualize dependencies or identify critical bottleneck courses affecting student progress.

Approach

Cleaned and standardized raw registrar data, then parsed prerequisite logic using regular expressions to extract structured relationships. Built a directed graph with NetworkX and created both static and interactive visualizations with Plotly. Developed a reusable Python API (demonstrated in `api/API_Demo.ipynb`) and a web-based UI prototype in the `webapi/` folder, so the pipeline can be applied to any MSU major.

Results & Impact

Delivered a working API and interactive curriculum graph covering CNS majors, with a prototype web UI for broader exploration. The tool enables analysis of course dependency chains and DFW rate patterns. Completed as a team capstone project with an MSU academic unit sponsor, with all code and reproducibility steps documented in the repository.

Educational Log Analysis Toolkit

Extended and maintain a Python toolkit for processing large educational log files from Concord Consortium tools. Added a new script to clean messy student text responses (removing HTML, encoding issues, and copied text), made the entire toolkit Mac-compatible, and keep scripts operational as file formats evolve.

Python pandas JSON/CSV parsing Regular expressions Command-line tools Cross-platform development (Windows → Mac)
More details

Problem

Educational platforms generate messy log files with nested JSON and unreadable student text responses (encoding errors, HTML tags, copied content). The existing toolkit was Windows-only and lacked text cleaning. Researchers needed a cross-platform solution to analyze genuine student writing.

Approach

Extended the existing Python toolkit by adding a student text response cleaner that detects and strips copied text from shared classroom tiles, removes HTML formatting, and normalizes encoding. Ported the entire toolkit from Windows-only to Mac-compatible. Maintain all scripts (date analysis, file splitting, JSON extraction, de-identification) to handle schema changes over multiple semesters.

Results & Impact

The text cleaner now automatically processes 100,000+ record datasets, outputting separate files for original vs. total student writing. Mac compatibility expanded the toolkit's user base to the whole research team. Maintained ~4x speedup over manual processing while enabling privacy-compliant data sharing.

Lean Analytics: Cycle Time & Delivery Performance

Built a suite of Python tools during a General Electric internship to analyze software delivery performance from Microsoft Teams Planner data. The tools calculate cycle times and on-time delivery rates to help engineering teams identify bottlenecks and improve throughput.

Python pandas matplotlib openpyxl
More details

Problem

GE engineering teams tracked tasks in Microsoft Teams Planner but had no easy way to extract lean metrics like cycle time or delivery delta from the raw export data. Without visibility into these numbers, teams struggled to identify bottlenecks during retrospectives.

Approach

Wrote scripts to ingest Teams Planner Excel exports and calculate cycle times for completed and in-progress tasks. Built a 3-pass deduplication algorithm to correctly match tasks across bucket transitions. Generated per-artifact and per-feature delivery charts, plus weekly stacked bar charts of on-time vs. late completions.

Results & Impact

Delivered visualizations and metrics used directly in engineering team retrospectives at GE. Tools were refactored post-internship for portability — hardcoded paths replaced with CLI arguments and logic modularized for reuse across any Teams Planner export.

NYC 311 Complaint Prediction

Built an AWS-based machine learning model to predict NYC 311 complaint categories and help operators route calls faster to the right agency. Identified and removed a key data leakage issue (the 'agency' field) that was creating artificially perfect 85% accuracy, achieving realistic 48.4% accuracy — 2x better than random guessing.

AWS (S3, Athena, SageMaker) Python scikit-learn pandas Logistic Regression Jupyter Notebooks
More details

Problem

NYC 311 operators receive thousands of calls daily and must manually determine which agency should handle each complaint (NYPD, HPD, DSNY, etc.). This manual routing takes time and leads to misrouting, especially for overnight calls when fewer staff are available.

Approach

Loaded 157,000+ 311 service requests from AWS S3 using a boto3 pipeline. Preprocessed data with pandas and scikit-learn, encoding categorical variables and scaling numerical features. Trained a multiclass Logistic Regression baseline model. Discovered that including the 'agency' column created unrealistic 85% accuracy (model was memorizing that HPD = Housing, DSNY = Sanitation), so removed it for realistic real-world predictions. Evaluated with accuracy, precision, recall, and confusion matrix.

Results & Impact

Final model achieves 48.4% accuracy without agency — 2x better than random guessing (25%). Correctly identifies Housing as the top complaint at 2am in residential Bronx zip codes (249 calls), directly answering the stakeholder's routing question. The model demonstrates that pure time/location data can provide useful routing suggestions even without knowing the agency upfront.
